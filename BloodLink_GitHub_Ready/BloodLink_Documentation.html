🩸 BloodLink

Complete Project Description & Technical Documentation

A Beginner-Friendly Guide to the Full-Stack Blood Bank Management System

1. Project Overview

BloodLink is a comprehensive software application designed to help hospitals and blood banks efficiently manage their blood supply. Instead of using paper records or simple spreadsheets, BloodLink uses a database and a secure website interface to handle operations.

💡 For Beginners: What is a "Full-Stack" Application?
A full-stack app has three parts: the "Frontend" (what the user sees and clicks), the "Backend" (the invisible brain processing rules), and the "Database" (where data is permanently stored). BloodLink controls all three.

The Problems BloodLink Solves:

  • Inventory Management: It knows exactly how many bags of each blood group (A+, O-, AB+, etc.) are in the fridge at any given moment.
  • Health Safety (56-Day Rule): Medical rules say a donor must wait 56 days between donations to stay healthy. The system calculates this automatically and blocks donations if a donor tries to give too early.
  • Hospital Requests: When a hospital requests blood, the system instantly checks the inventory to see if that blood is available, approving it or putting it on hold.
  • Security: Only authorized administrators can change stock or approve requests.

2. System Architecture

Architecture simply means how the different pieces of the software talk to each other. BloodLink is built in a 3-Tier Architecture.

Tier 1: Frontend (The UI the user clicks) [ index.html (Public) | admin.html (Admin) | styles.css ] ↓ HTTP Requests (JSON) ↓ Tier 2: Backend (The Java Brain) [ WebServer.java | HttpHandlers | AuthUtils ] ↓ JDBC (Database Link) ↓ Tier 3: Database (The Storage) [ SQLite Database File (bloodbank.db) ]

How it works together:

  1. You click a button on the website (Tier 1).
  2. The website sends a message to the Java server asking it to do something (Tier 2).
  3. Java checks its rules, connects to the SQLite Database (Tier 3), and saves or retrieves the data.
  4. Java sends the answer back to the website, which updates the screen for you.

💡 For Beginners: What is an ID / Primary Key?
If there are two donors named "John Doe", the computer needs a way to tell them apart. We give each person a unique "ID" number (like Donor #1 and Donor #2). This ensures data doesn't get meant for the wrong person.

5. The Java Backend (How it works under the hood)

WebServer.java (Main API Server)

This single file is the core of the whole backend application. It does two things:

  1. It opens port 9090 on your computer so browsers can connect.
  2. It creates 18 "Endpoints". An endpoint is basically a unique URL address that triggers a specific Java function. For example, visiting /api/donors runs the Java code to fetch the list of donors.

Examples of Endpoints:

  • /api/login - Checks username against the scrambled password.
  • /api/donations - Saves a new donation to history and adds +1 bag to the fridge.
  • /api/compatibility - Figures out if a patient can safely receive a specific blood bag.

DatabaseManager.java & AuthUtils.java

DatabaseManager.java acts as the "Bridge." Every time Java needs to read or save data, it asks DatabaseManager to pass the message to SQLite.

AuthUtils.java is the "Bouncer." When an admin creates a password like "admin123", AuthUtils scrambles it into a 64-character math string. If a hacker steals the database, they only see the scrambled math string, not the real password.

6. The Frontend (The User Interface)

1. Public Portal (index.html)

When you first load the page, you see the public portal. Anyone can view this. It shows live statistics (like how many total bags exist) to encourage people to donate. It also has a Donor Login and a button for Admins.

2. Admin Portal (admin.html)

This is hidden behind a password. It is a massive dashboard with 8 different tabs to run the business:

  • Dashboard: Visual charts of the blood stock and buttons to export PDF reports.
  • Donors: A list of everyone. It automatically shows a green "Ready" or a yellow wait timer.
  • New Donation: A form where admins type the Donor ID to record blood collected.
  • Requests: A screen showing hospital orders. Admins click "Fulfill" to approve them.
  • Activity Log: A security screen that shows every single button click made by any Admin.
💡 For Beginners: What is "Glassmorphism"?
If you notice the UI cards look like frosted glass with blurred backgrounds, this is a modern CSS design trend called "Glassmorphism" applied heavily in styles.css.

7. Step-by-Step Code Flows (How logic connects)

Scenario A: An Admin Records a Donation

  1. The admin looks at the website, types Donor ID #5, and clicks "Submit".
  2. JavaScript takes that ID and sends a message to Java: "Hey Java, Donor 5 just gave 1 bag."
  3. Java asks the Database: "When was the last time Donor 5 donated?"
  4. The Database answers: "They last donated 10 days ago."
  5. Java calculates: 10 days is less than the required 56 days. It stops the process!
  6. Java sends a message back to the website: "Error: Donor is not eligible. Must wait 46 more days."
  7. The website displays a red error warning the Admin.

Scenario B: A Hospital Requests Blood

  1. City Hospital asks for 5 bags of O- blood.
  2. Java asks the Database: "How many O- bags are in the fridge?"
  3. Database answers: "We only have 2 bags of O-."
  4. Java realizes there is not enough blood. It creates the order, but marks the status as "Pending (Low Stock)."
  5. The website shows a flashing Yellow Alert to warn the Admin that a hospital needs blood they don't have.
💡 For Beginners: Database Transactions
When a donation succeeds, Java must do two things: 1) Save the history record. 2) Update the fridge stock. If the internet crashes between step 1 and step 2, the data becomes broken. To prevent this, Java uses "Transactions". A Transaction says: "Either do EVERYTHING successfully, or cancel everything immediately. Never do just half the job."
