1. Introduction & Vision

What is BloodLink?
BloodLink is a comprehensive, multi-role digital ecosystem replacing outdated, manual blood banking systems. It bridges the critical gap between donors who want to help and patients in urgent need.

2. System Design Overview

We engineered a very strict Three-Tier Decoupled Architecture.

  • Tier 1 - Presentation: HTML5, CSS3, ES6 JavaScript. The "Face". Ensures a responsive, premium user experience.
  • Tier 2 - Application Logic: Java 17+. The "Brain". Handles security, routing, and medical calculations without external framework bloat.
  • Tier 3 - Persistence: SQLite 3. The "Vault". A file-based SQL database ensuring absolute relational data integrity.

3. Frontend Unveiled: Dynamic "Single-Page" Routing

When you use BloodLink, you notice that shifting between portals happens instantly. It never shows a white loading screen.

How is this engineered? CSS State Management.
  • The Illusion: Instead of having 5 separate HTML files, we coded every single dashboard into one massive file: index.html.
  • The Mechanism: Our JavaScript acts like a stage director. When a user logs in, the JavaScript checks their secure "Role Token."
  • The Execution: It immediately applies the CSS rule display: none; to every single section except the one the user needs to see.
  • The Benefit: Because all visual elements are already loaded into your browser's Random Access Memory (RAM), opening a new tab takes absolutely zero network time.

4. Java Server Unveiled: The "Traffic Controller"

How does the Java server know what to do when fifty different laptops are sending information at the same time?

The Answer: HTTP Context Registration
  • When the WebServer.java file boots up, it essentially sets up "toll booths" for every possible task. It uses server.createContext("/api/donors", new DonorsHandler()); to set up a specific booth.
  • If a request arrives at the /api/login booth, the Java server ignores everything else and hands that request exclusively to the LoginHandler.class.
  • This means our Java application is strictly compartmentalized. An error in fetching donors will never crash the login system.

5. Database Connectivity: The "Singleton Lock"

A fatal flaw in beginner applications is allowing the database to be opened too many times, which causes a "Database is Locked" crash.

Our Solution: The Synchronized Singleton Pattern
  • What is a Singleton? We force the entire Java server to share precisely one master connection to the SQLite vault via DatabaseManager.java.
  • The Keyword: We labeled the connection method with the Java keyword synchronized.
  • How it saves lives: Imagine three hospital admins click "Fulfill Request" for the very last bag of A+ blood at the exact same millisecond. Because the pipeline is synchronized, Java forces them into a queue (Admin 1, then Admin 2, then Admin 3) processed a millisecond apart. Admin 1 gets the blood, Admin 2 gets an "Insufficient Stock" error. No data corruption occurs.

7. The Power of Referential Integrity Constraints

In a beginner's view, "Referential Integrity" is the physical glue that keeps reality matching the database. It prevents "Ghost Data" from existing.

What is a constraint? Think of it as a strict bouncing guard at a club. The database simply will not allow an action if it breaks reality.
  • The Ghost Hospital Problem: Imagine a Blood_Request asks for 10 bags of A+ for "Hospital #55". What happens if someone accidentally deletes Hospital #55 from the system?
  • How Constraints protect us: We use a Foreign Key link. The database knows that the Request specifically belongs to the Hospital. If you try to delete the Hospital, the database screams "ERROR! You cannot delete this hospital because they have pending requests!" This prevents the loss of vital transaction histories.

Appendix: Core Code Explanations

Direct breakdowns of the critical source code algorithms and structural logic.

A. PreparedStatement: Defending Against Hackers

(Found inside Java Handlers like LoginHandler.java)

// Java Server
String query = "SELECT role FROM Users WHERE username = ?";
PreparedStatement pstmt = conn.prepareStatement(query);
pstmt.setString(1, reqUsername);
ResultSet rs = pstmt.executeQuery();
  • The Explanation: We intentionally used PreparedStatement (the ? in the code) instead of a regular text string.
  • Why? This absolutely blocks SQL Injection Attacks. If a hacker types admin' OR 1=1; DROP TABLE Users; to delete our database, the PreparedStatement treats it strictly as pure text (a username), not as an executable command fragment. Without this, the database is severely vulnerable.

B. The ACID Transaction Code: Preventing Lost Data

(Found in RequestsHandler.java)

// Java Server
conn.setAutoCommit(false); // Step 1: Pause automatic saving

// Step 2: Deduct Inventory
String updateInv = "UPDATE Inventory SET qty = qty - ? WHERE type = ?";
// ...execute updateInv

// Step 3: Change Request Status to 'Approved' 
String updateReq = "UPDATE Requests SET status = 'Approved' WHERE id = ?";
// ...execute updateReq

conn.commit(); // Step 4: Save EVERYTHING safely at the exact same time
  • The Explanation: This is our Atomicity (All-or-Nothing) Engine. By setting setAutoCommit(false), we tell SQLite, "Do not save anything until I say so."
  • The code deducts inventory and updates the patient status. Only if both steps succeed completely do we call commit(). If Step 3 triggers an error, the system automatically rolls back, and the blood returns to the vault.

D. Data Display Unification (COALESCE)

(Found in UsersHandler.java for the Admin Portal)

-- SQLite Database Query
SELECT u.username, u.role, 
       COALESCE(d.name, h.name, r.name, 'Admin User') AS display_name
FROM Users u
LEFT JOIN Donors d ON u.donor_id = d.donor_id
// ... other JOINs
  • The Explanation: Because our database uses Third Normal Form (3NF) separation, Names are strictly separated into 3 different tables (Donors, Hospitals, Requesters).
  • The SQL COALESCE function solves this cleanly. It grabs the very first 'Name' it finds that is NOT empty across those three tables, and re-packages it cleanly as display_name for the Admin dashboard.

E. Raw Byte Input Streams: Reading HTTP Data

(Found inside all HTTP Handlers in WebServer.java)

// Java Server - Reading incoming data
InputStream is = exchange.getRequestBody();
String requestBody = new String(is.readAllBytes(), StandardCharsets.UTF_8);
  • The Explanation: Because we built the server from scratch (without frameworks like Spring Boot), Java receives web traffic as raw binary data (Bits & Bytes).
  • This code snippet manually catches the incoming byte stream from the Javascript Fetch API, and securely translates those bytes back into a readable UTF-8 String containing the JSON sent from the browser.

F. Cross-Origin Security Headers (CORS)

(Found at the top of HTTP Handlers in WebServer.java)

// Java Server - HTTP Policy Definition
exchange.getResponseHeaders().add("Access-Control-Allow-Origin", "*");
exchange.getResponseHeaders().add("Content-Type", "application/json");
  • The Explanation: This is a strict network safety policy. Browsers normally block websites from randomly talking to external servers to prevent cross-site scripting attacks.
  • By adding Access-Control-Allow-Origin: *, we tell the user's browser, "We are the server, and we are explicitly permitting this UI to retrieve our JSON data." This ensures the API handshake completes successfully.

G. Byte to Hex Conversion: Generating the Hash

(Found inside AuthUtils.java)

// Java Server - SHA-256 Math
MessageDigest digest = MessageDigest.getInstance("SHA-256");
byte[] encodedHash = digest.digest(password.getBytes(StandardCharsets.UTF_8));

StringBuilder hexString = new StringBuilder(2 * encodedHash.length);
for (byte b : encodedHash) {
    String hex = Integer.toHexString(0xff & b);
    if(hex.length() == 1) hexString.append('0');
    hexString.append(hex);
}
  • The Explanation: This code executes the SHA-256 security routine. The MessageDigest chops the password into algorithmic bytes.
  • However, raw bytes aren't readable text. The for loop iterates over every single mathematical byte, grabs its low 8 bits (0xff & b), and converts it into a Hexadecimal string (numbers 0-9 and letters A-F) so we can easily store the 64-character hash securely in the SQLite database.

H. Dictionary State Matrix: O(1) Time Complexity

(Found inside Javascript in index.html)

// JavaScript Frontend - Compatibility Engine
const bloodCompatibility = {
    "O-": ["O-"],
    "O+": ["O+", "O-"],
    "A-": ["A-", "O-"],
    // ... other types
    "AB+": ["O-", "O+", "A-", "A+", "B-", "B+", "AB-", "AB+"]
};

// Checking instantly
let compatibleTypes = bloodCompatibility[patientType];
  • The Explanation: Instead of writing 20 complicated If/Else statements to figure out who can receive what blood, we wrote a Key-Value Dictionary Object (A Map).
  • This means when a patient logs in as "O+", Javascript looks up "O+" as a key and instantly gets the array `["O+", "O-"]`. In algorithmic terms, this retrieves medical safety data in O(1) Time Complexity, meaning it requires exactly 1 step to process, maximizing speed.
