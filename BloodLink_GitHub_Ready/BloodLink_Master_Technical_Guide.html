The Blueprint: Database Schema

The foundation of the system is the SQLite database. Below is the full relational schema and a modular data map for each component.

CREATE TABLE IF NOT EXISTS Donors (
    donor_id INTEGER PRIMARY KEY AUTOINCREMENT,
    name TEXT NOT NULL,
    blood_group TEXT NOT NULL,
    contact_number TEXT,
    email TEXT,
    address TEXT,
    last_donation_date DATE
);

CREATE TABLE IF NOT EXISTS Blood_Inventory (
    blood_group TEXT PRIMARY KEY,
    quantity_bags INTEGER DEFAULT 0
);

CREATE TABLE IF NOT EXISTS Hospitals (
    hospital_id INTEGER PRIMARY KEY AUTOINCREMENT,
    name TEXT NOT NULL,
    contact_number TEXT,
    address TEXT
);

CREATE TABLE IF NOT EXISTS Blood_Requests (
    request_id INTEGER PRIMARY KEY AUTOINCREMENT,
    hospital_id INTEGER,
    requester_id INTEGER,
    blood_group_required TEXT NOT NULL,
    quantity_needed INTEGER NOT NULL,
    request_date DATE,
    status TEXT,
    is_emergency INTEGER DEFAULT 0,
    FOREIGN KEY (hospital_id) REFERENCES Hospitals(hospital_id)
);

CREATE TABLE IF NOT EXISTS Users (
    user_id INTEGER PRIMARY KEY AUTOINCREMENT,
    username TEXT UNIQUE NOT NULL,
    password_hash TEXT NOT NULL,
    role TEXT DEFAULT 'Admin',
    donor_id INTEGER,
    hospital_id INTEGER
);

CREATE TABLE IF NOT EXISTS Activity_Log (
    log_id INTEGER PRIMARY KEY AUTOINCREMENT,
    action TEXT NOT NULL,
    description TEXT,
    timestamp DATETIME DEFAULT CURRENT_TIMESTAMP
);

WebServer.java: Complete Breakdown

The WebServer.java file is the most critical part of the application. It acts as the API Server that connects your website and your database.

1. Routing & Context Initialization

The main method defines the entry points (Endpoints) for the web application. When you go to a URL like /api/login, the server knows exactly which "Handler" to wake up.

Technical Note Uses com.sun.net.httpserver for a lightweight, zero-dependency server.

2. Full Authentication & Register Flow

Handlers: LoginHandler and RegisterHandler.

  • Hashing: Passwords are hashed using SHA-256 before being compared to the database.
  • Transactions: Registration uses conn.setAutoCommit(false). This ensures that if creating a Donor succeeds but creating their User login fails, the system rolls everything back to prevent broken data.

3. The 56-Day Rule (Donor Module)

Handler: DonationsHandler.

Before allowing a donation, the backend runs a SQL calculation using julianday(). If today - last_donation < 56, the request is rejected with a 400 Error and a "Not Eligible" message.

DatabaseManager.java: The Connector

This class manages the lifecycle of the database connection. It ensures that the database is optimized and WAL (Write-Ahead Logging) is enabled for better speed.

package org.bloodbank;
import java.sql.*;

public class DatabaseManager {
    private static final String URL = "jdbc:sqlite:bloodbank.db";

    public static Connection getConnection() throws SQLException {
        Connection conn = DriverManager.getConnection(URL);
        Statement s = conn.createStatement();
        s.execute("PRAGMA journal_mode=WAL"); // Multi-user speed boost
        return conn;
    }

9. The Comprehensive System Workflow

This section details the end-to-end technical life-cycle of data passing through the system, from the initial server launch to the delivery of blood to a hospital.

A. System Boot & Environment Setup

When the Java application is executed, the Main.java entry point triggers three critical processes:

  1. Persistence Check: The DatabaseManager initializes the SQLite connection. It enables PRAGMA journal_mode=WAL, which is essential for "Write-Ahead Logging." This allows the Web Server and Desktop GUI to perform concurrent operations without locking the database file.
  2. Schema Enforcement: The server reads schema.sql and runs CREATE TABLE IF NOT EXISTS commands. This ensures that the system can recover even if the database file is missing or corrupted.
  3. Interface Binding: The WebServer binds to Port 9091 and initializes the HttpServer with 18 distinct API contexts.

B. The Donor Lifecycle (Supply Chain)

How and when blood enters the repository is strictly controlled by software-defined medical rules:

  • Creation: A donor registers via the RegisterHandler. A dual-entry is created: one in Donors (medical profile) and one in Users (security profile).
  • Eligibility Guard: When a donor attempts to give blood, the DonationsHandler queries the database using the Julianday function. This prevents human error by mathematically proving that at least 56 days have passed since the last entry.
  • Inventory Increment: Upon a successful donation, a Database Transaction ensures that the Donation_Records are updated simultaneously with the Blood_Inventory. If one fails, the other is rolled back (Atomic Operation).

C. The Request & Fulfillment Lifecycle (Demand Chain)

The system treats every request as a state-managed entity:

  1. Submission: A patient or hospital submits a request via /api/requests.
  2. Real-Time Validation: The server queries the Blood_Inventory table for the specific blood group.
  3. Status Assignment:
    • Automatic Approval: If Stock >= Request, the status is set to "Approved" and stock is immediately subtracted.
    • Pending State: If stock is low, the request is stored as "Pending (Low Stock)." It is then highlighted in red on the Admin Dashboard for manual intervention.
    • Emergency Handling: Requests marked for emergency bypass specific non-critical checks to ensure life-saving speed.

D. The Audit & Monitoring Core

Because this is a medical application, accountability is paramount:

  • Audit Trail: Every destructive action (e.g., deleting a donor or record) is caught by the handlers and passed to the logActivity() method. This records the action, actor, and precise microsecond in the Activity_Log.
  • State Aggregation: The StatsHandler provides the "Pulse" of the system. It runs aggregate queries (SUM, COUNT) so that managers can see real-time trends instead of raw table data.

Technical Data Flow Logic

  1. Supply Path (Input):
    • Donor Profiling: Data is ingested into the Donors table.
    • Unit Registration: A physical bag of blood is registered in Donation_Records and linked to the donor's ID.
    • Inventory Influx: An Atomic SQL Update increments the specific group count in the Blood_Inventory table to maintain real-time accuracy.
  2. Demand Path (Output):
    • Submission: Hospital or patient requests are stored in the Blood_Requests table.
    • Inventory Check: The backend performs a lookup in the Blood_Inventory table for available units.
    • State Transition: If approved, an Atomic SQL Update subtracts from the inventory and sets the status to "Approved."
  3. Security & Audit Trail:
    • Accountability: Every state transition (Registration, Donation, Approval) is permanently recorded in the Activity_Log.
    • Relational Constraints: Foreign keys prevent the deletion of a donor who has historic donation records, preserving data integrity for medical history.
